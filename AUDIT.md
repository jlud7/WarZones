# WarZones Audit

**Auditor:** senior full-stack game dev / product designer
**Scope:** full repo at `main` (`b00bb3a`) вҖ” `index.html`, `script.js` (7076 LOC), `style.css` (3446 LOC), `generate-portraits.py`, `.github/workflows/generate-portraits.yml`, `README.md`. No `package.json`, no build, no tests, no Firebase rules file.
**Assumptions made:**

- The `apiKey` in `index.html:18` literally contains the word `"placeholder"` вҖ” assumed the production key is stored elsewhere and this repo is not actively connected to live Firebase. Findings that depend on live config are flagged as **if-live**.
- No `database.rules.json` / `firestore.rules` is in the repo. Assumed rules are either default test-mode (world-readable for 30 days) or whatever was last edited in the Firebase console. **This is the single biggest unknown in the audit.**
- Target audience assumed to be casual web players, probably on mobile, coming via the CodePen / GitHub Pages link.

---

## Executive Summary

WarZones is a charming, visibly loved single-author vanilla JS game with a real creative core (four-layer Battleship + power-ups + a surprisingly well-written single-player campaign with 10 missions, portraits generated by a GitHub Action calling Replicate). The campaign is the best part of the codebase вҖ” data-driven, modifier-based, and shares the same rules engine as skirmish mode. **The biggest weakness is the online multiplayer, which is not just unsafe вҖ” it is trivially cheatable, by design**: each client is authoritative over its own board, ships are transmitted in full at the start of combat, and the opponent's client trusts every attack result it receives. A mildly technical player with DevTools open wins every online match without detection. The second biggest weakness is that 7076 lines of game code, UI code, network code, and sound synthesis live in a single `script.js` with two duplicate method definitions, four calls to a method that doesn't exist (`ui.showMainMenu`), and a touchmove handler that rotates ships on every horizontal swipe вҖ” including intended scrolls.

Headline priority: **fix the online-mode trust model before the next visible launch, or disable the Play Online button.**

## Findings by Category

Tags: рҹ”ҙ Critical В· рҹҹ  Major В· рҹҹЎ Minor В· рҹ”ө Nitpick. Where I say "taste," it's taste; where I say "defect," it's a defect.

### 1. Architecture & Code Structure

**Strengths:**

- `GameState` (`script.js:3619вҖ“3990`) is decently isolated: `placeShip`, `processAttack`, `checkGameOver`, `isValidPlacement`, `calculateShipPositions` are mostly pure and don't touch the DOM. Campaign's modifier hooks (`beforePlayerAttack`, `afterPlayerMiss`, etc.) compose cleanly with `processAttack`.
- The class list (GameState, GameAI, UIController, SoundManager, AnimationManager, Statistics, CampaignManager, NetworkManager) is the *right* decomposition вҖ” it just all happens to live in one file.

**Issues:**

- рҹ”ҙ **God file.** 7076 lines of game logic, UI orchestration, sound synthesis, Firebase integration, and a full 10-mission campaign in a single flat script with no modules. The class `WarZones` alone is ~2500 lines and is both the controller and half the UI glue. Directly responsible for findings #5, #6, #8. Fix: ES modules (Top-10 #3). No bundler required вҖ” `<script type="module">`.
- рҹҹ  **State is entangled with rendering.** `handleAttack` (`script.js:2542вҖ“2772`) does `processAttack`, then plays sound, updates the DOM, triggers animations, updates commentary, updates scoreboard, checks game over, and switches turn all in one function. Win: extract to `resolveAttack(result)` (state-only) then render separately. Pays off when adding replay or AI-vs-AI modes.
- рҹҹ  **Dual sources of truth.** Ship state is stored in both `ships.{side}.{type}.{positions, hits, isSunk}` and mirrored as literal strings (`"hit"`, `"miss"`, ship-type names) inside `boards.{side}.{layer}[idx]`. Several places read one, several read the other; at least one defensive patch (`script.js:1810вҖ“1828`) exists because Firebase strips empty arrays on round-trip and so `ship.hits` has to be rehydrated. Single source of truth recommended.
- рҹҹ  **Dead code pretending to be a feature.** `loadSavedGame` without a paired save (Top-10 #8). `UIController.showSonarEffect` (`script.js:5243`) still exists though SonarPulse was renamed to CannonBall. `debounce` and `calculateAccuracy` at the bottom of the file are unreferenced. The `Tutorial` class is instantiated (`script.js:55`) but `start()` is never called.
- рҹҹЎ **Comments contain change-log narration.** Many comments describe fixes to past bugs ("Fix 3:", "Without this a fast player could fire the same ATTACK twice"). These belong in commit messages; they rot as the code evolves.
- рҹҹЎ **`// removed` / renamed-comment debt.** `CannonBall (Renamed from SonarPulse)` at `:39` is fine, but `showSonarEffect` and `AI recordShipDetection` (`:4077`) are leftover surface from that rename.

### 2. Game Logic & Rules Engine

**Strengths:**

- `processAttack` (`script.js:3831вҖ“3908`) is a clean reducer-style function: input `(boardId, index, layer)`, output `{hit, sunk, treasure, mine, shipType, gameOver, ...}`. Exactly the shape a testable rules core needs.
- Campaign modifiers hook *around* the engine instead of forking it вҖ” `beforePlayerAttack` (`:6225`) returns `{blocked, reason}`, `afterPlayerMiss/Hit` records state for fog/decay. This is genuinely good design.
- `moveHistory` (`:3679`) records both placements and attacks with enough data to replay. Unused, but latent.

**Issues:**

- рҹҹ  **Phase state machine is implicit.** `gameState.phase` is a free-form string (`'setup' | 'combat' | 'gameOver'`). Transitions are scattered across `startCombatPhase`, `_initCombat`, `showGameOver`, `handleIncomingAttack`, `handleAttackResult`, `startOnlineCombat`, `launchMission`. Several places write `phase = 'gameOver'` independently. A formal state machine (`{setup вҶ’ placement вҶ’ combat вҶ’ gameOver}`, with explicit transitions) would catch the edge cases in Top-10 #4 and #6.
- рҹҹ  **No determinism.** Ship placement (`placeAIShips`, `_placeExtraShips`, `_placeMines`), treasure placement (`placeTreasureChests`), AI personality (`:4026`), AI targeting (everywhere) all call `Math.random()` directly. A replay from `moveHistory` would not reproduce. If you ever want shareable replays or deterministic bug repros, thread a seeded PRNG through (`mulberry32` is 6 lines).
- рҹҹ  **Attack undo is wired but dangerous.** `undoLastMove` (`:3935`) handles `type: 'attack'` вҖ” but during combat the only UI affordance is "Undo Placement", and no combat code calls it. A refactor that exposes it would corrupt turn flow, win counters, and AI memory (`this.ai.attackedPositions`). Recommendation: delete the attack branch; state the invariant "undo is placement-only."
- рҹҹЎ **Win check is O(ships Г— hits) every attack.** Fine at this scale (5вҖ“6 ships, вүӨ5 cells each), flagging in case the ship roster grows.
- рҹҹЎ **`isValidPlacement` doesn't reject off-board first cell**, it relies on `calculateShipPositions` returning `[]`. Works, but brittle вҖ” the square-shape branch has an implicit `col <= boardSize - 2 && row <= boardSize - 2` check that would silently misbehave if `BOARD_SIZE` ever changed. Move the bounds check up.

### 3. Firebase & Multiplayer (highest priority)

This is where the audit earns its keep.

**Strengths:**

- The message-log pattern (`/rooms/{code}/messages` with `child_added` listener) is the right primitive for turn-based games.
- Pre-marking existing message IDs on join (`_onRoomPaired`, `:6849вҖ“6858`) with a clear comment explaining the server-clock skew bug that motivated it вҖ” good defensive engineering.
- `onDisconnect().remove()` on the host (`:6746`) cleans rooms; guest cleans its own node. Reasonable.
- `_sanitize()` (`:6909`) strips `undefined` before `push()` (Firebase rejects otherwise). Good; the comment about empty arrays being dropped is honest about the workaround.
- 7-char room codes from a 30-char alphabet (`:6665`) вүҲ 22 billion combinations, with confusable characters excluded. Good room-code UX.

**Issues:**

- рҹ”ҙ **Ship positions are transmitted to the opponent.** `startCombatPhase` вҶ’ `network.send({type: 'SHIPS_READY', ships: this.gameState.ships.player})` (`:2372`). The message is a full `{shipType: {positions, hits, isSunk}}` object. `loadOpponentShips` (`:1804`) normalizes it and stores positions into `gameState.ships.opponent` and populates `boards.opponent`. Any cheater can (a) read the message from Firebase directly or (b) inspect `warZonesGame.gameState.ships.opponent` in DevTools. **This is the defining multiplayer vulnerability.** Fix requires restructuring the protocol (Top-10 #1) вҖ” keep ships private, have the defender resolve attacks against its own state, and send back only hit/miss/sunk flags + remaining-ship counts.
- рҹ”ҙ **Client is fully authoritative over results.** `handleAttackResult` (`:1896`) trusts `result.hit`, `result.sunk`, `result.shipType`, `result.gameOver` from the peer and updates local state accordingly. There *is* a local fallback `checkGameOver()` (`:1935`) вҖ” good вҖ” but nothing verifies the opponent's hit/miss honesty. A cheater replies "miss" to every `ATTACK` and is effectively invincible.
- рҹ”ҙ **No turn authority.** `currentTurn` is tracked locally on each client; incoming `ATTACK` messages are processed regardless of whose turn it should be. A cheater can skip opponent turns.
- рҹ”ҙ **No Firebase security rules in repo.** If rules are default test-mode or unconfigured, any holder of a room code can read *all* messages (including SHIPS_READY) and write any message as any sender (`from: this.clientId` is a client-chosen string). This compounds with the other three.
- рҹҹ  **No authentication.** `clientId = Math.random().toString(36).slice(2, 10)` (`:6632`) is a browser session identifier, not a user identity. Any field that relies on it (e.g. skipping own messages at `:6870`) can be trivially spoofed by an attacker who reads both IDs out of `/rooms/{code}`.
- рҹҹ  **Messages append forever.** `/rooms/{code}/messages` grows unboundedly during a match; the whole history is kept until host disconnects. For a 30-move match Г— multiple attack/result/ping/pong, ~100+ messages. Only the host-disconnect `remove()` ever clears. Add a message TTL / prune rule.
- рҹҹ  **Listener leak surface.** `otherRoleListener`, `hostDisconnectListener`, `messagesListener` are all re-registered per connect; `reset()` tries to detach them (`:6955вҖ“6997`), but `reset()` isn't called on every tear-down path (e.g. `cancelHost` calls `reset()` вҖ” good; but back-to-menu mid-match via the Escape menu does not, it just toggles display). Map reconnection scenarios and audit.
- рҹҹ  **Compat SDK.** `firebase-app-compat.js` + `firebase-database-compat.js` at 9.23.0 adds ~200 KB gzipped. Modular SDK v10+ imports only what you use. If you're serving from GitHub Pages with no caching strategy, every first visit pulls the full bundle.
- рҹҹЎ **Placeholder API key in the HTML.** `AIzaSyD_placeholder_warzones_a515c` in `index.html:18` is not a valid Google API key format (not 39 chars, contains underscores/literals). If this is what's deployed, the whole online flow is broken before any of the above matters; if this is a scrub and the real key lives in a branch/environment, say so in the README. Either way: the current repo can't be used to *test* online mode end-to-end.
- рҹҹЎ **No reconnect / rejoin.** If either player's tab crashes, the room is gone (host cleanup) or unresponsive (guest). No "resume match" flow. For a fast-paced 2-player format this is probably acceptable, worth a product call.
- рҹҹЎ **Matchmaking is code-sharing only.** Fine for now; don't build a lobby system until you have a reason. But know it's a ceiling on discoverability.

**Recommended fix sketch (protocol):**

```
Host (my side)                        Guest (opponent)
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ                         в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
[ships stored locally, never sent]    [ships stored locally, never sent]

send: {type:'READY'}                  recv: READY                  вҶҗв”Җв”Җв”Җв”Җв”Җв”җ
recv: READY                           send: {type:'READY'}         в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ both sides confirm

send: {type:'ATTACK', idx, layer, seq} в”Җв”Җв”Җв”Җв”Җв–ә
                                      resolve against own ships
                                      в—„в”Җв”Җв”Җв”Җв”Җ {type:'RESULT', seq, hit, sunk, shipType?, remaining}

if remaining.mine === 0:              if remaining.opponent === 0:
  show 'Defeat'                         show 'Victory'
```

Guardrails that need rules enforcement: `seq` monotonic per-player; `from` field writable only by session-matching auth UID (or a HMAC over `clientId+roomCode+secret`); payloads limited to `<= 256 bytes`; writes to `/rooms/{code}` only by participants listed under `/rooms/{code}/host` or `/guest`.

### 4. Campaign Mode

**Strengths:**

- рҹҹў Easily the best-designed part of the codebase. 10 missions in 3 acts, data-driven (`_defineMissions()` at `:5730`), with modifiers (`fog_of_war`, `layer_fog`, `turn_timer`, `enemy_reinforcements`, `shields`, `hit_decay`, `mines`, `reduced_fleet`) composed into a final boss. Escalation curve is genuinely thoughtful: Hydra adds a ship, Kraken removes one of yours AND adds shields, Voss stacks four modifiers.
- рҹҹў Shares the same `processAttack` engine вҖ” no divergence. Modifier hooks are clean.
- рҹҹў Portraits-by-prompt via Replicate in a GitHub Action (`.github/workflows/generate-portraits.yml`) is a lovely, weird detail. The prompts are *character* prompts (names like "The Hydra", "The Kraken", "Admiral Voss") вҖ” the work of someone who cares about the game as a game, not a demo.
- рҹҹў Star-based scoring (three-star threshold per mission based on accuracy) and unlock gating via `highestUnlocked` вҖ” standard, solid.

**Issues:**

- рҹҹ  **Missions are hard-coded in `script.js`.** At 146 lines of mission data (`:5730вҖ“5876`), this should be `campaign-missions.json` loaded at startup. Makes it editable without touching code, enables future A/B-ing difficulty, and lets you add missions without a deploy. Taste-adjacent but the payoff is real.
- рҹҹЎ **Mine placement is Sea-layer only.** `_placeMines` (`:6204`) hard-codes `const layer = 'Sea'`. If the design intent is multi-layer mines, fix; if it's intentional, document in the mission config (`mineLayer: 'Sea'`).
- рҹҹЎ **Enemy name/portrait wiring is brittle.** `launchMission` (`:6088`) uses a string template containing `mission.isBoss ? 'вҳ пёҸ' : 'рҹӨ–'` (`:6121`) вҖ” if `img` errors, `onerror` replaces with a *literal text node of the string*. Works but ugly; prefer a proper fallback element.
- рҹ”ө **Difficulty tuning:** the only AI knob exposed is `unpredictability` + `clusterPreference`. For the late bosses you want more dimensions: ship hunt persistence, treasure priority, powerup strategy. Room to grow.

### 5. Code Quality

**Issues:**

- рҹ”ҙ **Duplicate method definition:** `activatePowerup()` at `:84` and `:798` вҖ” Top-10 #5.
- рҹ”ҙ **Missing method called four times:** `ui.showMainMenu()` at `:6763, 6793, 6818, 6896` вҖ” Top-10 #4.
- рҹҹ  **God methods.** `handleAttack` (`:2542`, 230 LOC), `activateCannonBall` (`:423`, 260 LOC), `_initCombat` (`:2444`), `showHowToPlay` (`:1491`, 150-line innerHTML string). Extract.
- рҹҹ  **Monkey-patching event-listener removal.** `activateBlackBox` (`:185вҖ“194`) and `activateCannonBall` (`:676вҖ“682`) wrap `removeEventListener` on a DOM element to chain their own cleanup. This is the code equivalent of a self-reference memory leak вҖ” any other listener added to those boards has the patched wrapper for its entire lifetime, and one element now has different `removeEventListener` semantics than its siblings. Replace with a single `{once: true}` listener + `AbortController` for multi-listener cleanup.
- рҹҹ  **Nested `setTimeout` chains drive flow control.** AI turn at `:2803`/`2918`, cannon-ball cleanup at `:751`, phase transition at `:2441`, treasure menu at `:2841`. Some are tracked in `aiTurnTimeouts` for cancel; many are not. On rapid "New Game" clicks, orphaned timeouts can still fire into a reset state. Put everything through a single cancellable scheduler.
- рҹҹ  **No error boundaries.** `handlePeerData` wraps in `try/catch` to reset `isProcessingTurn` if a handler crashes (`:1747, 1798`) вҖ” good defensive thinking after presumably fixing a real bug вҖ” but there's no other place that does this. If `placeAIShips` hits its 100-attempt ceiling and returns without placing a ship (`:2525`), the ship just silently isn't there; game will hang in combat.
- рҹҹЎ **Comment noise from prior fixes.** `"Fix 3: Modify startNewGame to clear all AI timeouts"` (`:1642`), `"Fix: ..."`, `"IMPORTANT ADDITION: ..."` вҖ” read like commit messages in-line. See category 1.
- рҹҹЎ **Global `window.warZonesGame`** assigned at `:72` so `GameAI.shipCompleted` can fall back to it (`:4941`). Should be an explicit dependency, not a global. You already pass `this` to the AI at construction вҖ” use it.
- рҹ”ө Inconsistent casing / style in method names (`uiUpdateForHumanPlacement` vs `updateUIForPlayerTurn`). Taste.

### 6. UI Design

**Strengths:**

- Tokens exist in `:root` (`style.css:1вҖ“28`) for color, cell size, border radius, shadow. Layer palette (space / sky / sea / sub) is distinct and consistent.
- Orbitron font + dark navy/red accent is a coherent "military sci-fi" vibe вҖ” distinctive enough, not generic. Doesn't look AI-generated.
- Placement fleet dock, ship counter, phase transitions, explosion particles, splash rings, confetti, and the AI "scanning" line all punch above weight for a hand-rolled vanilla game.
- Campaign portraits, boss tags, act headers, star ratings вҖ” the campaign *does* evoke its naval sci-fi reference deliberately.

**Issues:**

- рҹҹ  **Taste/defect boundary: Battleship reference is accidental, not deliberate.** The core skirmish UI (blue boards, ship emojis, grey background) looks more like "generic browser game" than "vintage Battleship". The campaign looks like it has its own art direction. If the pitch is "Battleship with twists", a tighter visual callback (paper/cardboard grid, peg-style hit/miss pieces, naval chart typography) would sharpen identity. If the pitch is naval sci-fi, lean further in and drop the pastel blue defaults. Pick one.
- рҹҹ  **Design system leaks everywhere.** 45 `!important`s, 44 `@keyframes`, 16 `@media` queries in one stylesheet. The `!important` count is the telltale sign вҖ” it means overrides are fighting specificity. You can collapse вүҘ half of the animations and most `!important`s by moving to CSS variables + a single source of truth for state classes.
- рҹҹЎ **Mixed font stack in overlays.** Orbitron for game chrome (good), but overlays like debriefings fall back to `-apple-system, BlinkMacSystemFont, 'Segoe UI'` (`style.css:1642, 1688, 1733, 1768, 1830`). Intentional for legibility? If so, document. If not, pick one.
- рҹҹЎ **Background is a data-URL SVG** (`style.css:53`) вҖ” fine, but it's been pasted inline and is hard to iterate on. Externalize.
- рҹ”ө **Microdetail:** hit/miss emojis (`рҹ’Ҙ` / `O`) are serviceable but inconsistent вҖ” emoji vs ASCII letter. Either both emojis (рҹ’Ҙ/рҹ’Ё) or both symbols. Taste.

### 7. UX & Interaction

**Strengths:**

- Fleet dock during placement, with placed/current/upcoming states, is a good signpost.
- "Hit? Fire again" is reinforced in commentary and works the same across AI, human, and online modes.
- Keyboard nav (arrow keys + 1-4 layer + space fire + R rotate + Esc menu + Ctrl/Cmd+Z undo) is actually strong for a web game. Keyboard hint bar at `:1122` is a nice touch.
- Copy-to-clipboard on the room code (`:1471`) вҖ” small, considerate.

**Issues:**

- рҹ”ҙ **Mobile touchmove always rotates.** Top-10 #6.
- рҹҹ  **Loading / waiting-for-opponent state is a single line of text.** "Waiting for opponent..." then silence. A host that has shared their code has nothing to do for 30вҖ“120 seconds and no feedback that the code is valid, that Firebase is healthy, or that the room exists. Add: room-code validation echo, a "tap to retry", a simple spinner.
- рҹҹ  **"Your turn" visibility in online mode.** When opponent is acting, the only feedback is commentary text. Add a dimmed opponent-board overlay + "Waiting for opponent..." pill while `currentTurn !== myPlayerId`. Currently a player can sit clicking helpfully and hear "It's not your turn!" in commentary.
- рҹҹ  **No confirmation before destructive moves.** "Reset Progress" on the campaign map does confirm (`:5988`). Good. "New Game" during a live online match does not вҖ” it silently orphans the opponent. Protect.
- рҹҹ  **No onboarding for twists.** "How to Play" (`showHowToPlay`, `:1491`) covers the four layers, ships, combat, and treasure. Good. But the twists are *the product*: treasure chests вҶ’ 3 powerups, 4-layer coordination, the Krypton Laser cross-layer attack. New players meet these only if they happen to find a treasure. Consider: a guided first-mission tutorial in campaign (Mission 1 "First Contact" is positioned for this).
- рҹҹЎ **Empty/error states in online setup are thin.** "Room not found" / "Could not join room" show for 2.5s then bounce back to a broken menu (because of the `showMainMenu` bug). Top-10 #4 also fixes this.
- рҹҹЎ **Treasure menu disables BlackBox when no empty sky cells exist** (`:5152`). Good. But the other two powerups could also be situationally useless (e.g., KryptonLaser when only Sky layer has ships left, if you hit a cleared cell in all other layers) вҖ” the menu doesn't reflect that.
- рҹ”ө Commentary box is small and disappears quickly. A transient log (last 3 events) would help players rebuild context after a phone call / alt-tab.

### 8. Accessibility

**Issues:**

- рҹ”ҙ `maximum-scale=1.0, user-scalable=no` (`index.html:5`) blocks pinch-zoom вҖ” violates WCAG 2.1 SC 1.4.4.
- рҹҹ  **Board cells are `<div>`.** 128 interactive cells generated in `createGameBoards` (`:1286вҖ“1399`) with `class="cell"`, `data-index`, `data-layer` вҖ” no `role`, no `aria-label`, no `tabindex`. A screen reader sees 128 empty generic elements. Keyboard nav DOES work (handled at `:1139`), but a screen-reader user has no way to know position A4 was a hit.
- рҹҹ  **Color-first state.** `.hit`, `.miss`, `.sunk`, `.fogged`, `.decayed`, `.shield-deflect` all rely on color + a small emoji for distinction. For colorblind users the emoji rescues the basic hit/miss distinction; decay/fog/shield are harder to tell apart by shape.
- рҹҹ  **No `prefers-reduced-motion` guard.** 44 keyframes all run. For users with vestibular sensitivity, the screen shake on hits is a problem.
- рҹҹЎ **Focus management.** When a modal opens (treasure menu, game over, how-to-play, briefing), focus isn't trapped; Escape closes the game menu but not overlays. Implement focus trap + Escape-to-close consistently.
- рҹҹЎ **Heading order** is mostly sensible (`<h1>`, `<h2>`) but `index.html` has no landmarks (`<main>`, `<nav>`). Add them.

### 9. Performance

**Strengths:**

- No framework, no bundler, no virtual DOM overhead вҖ” game is fundamentally cheap.
- Small board (4Г—4 Г— 4 layers = 64 cells total) means any render strategy is fine.
- Scoped animations with RAF (`playExplosion` `:3325`, `playConfetti` `:3471`) вҖ” reasonable.

**Issues:**

- рҹҹ  **Full opponent-board clone on combat start.** `_initCombat` (`:2468`): `board.parentNode.replaceChild(clone, board)` for each opponent layer to "force CSS animations to restart", then re-attaches click listeners. This is a workaround for not controlling animation state via classes. Replace with `el.classList.remove('foo'); void el.offsetWidth; el.classList.add('foo')` (already used at `:3441` correctly).
- рҹҹ  **No minification, no compression guarantee.** `script.js` is 244 KB uncompressed, `style.css` 73 KB uncompressed. GitHub Pages serves gzipped (~80 KB / ~15 KB) but a one-shot `esbuild --minify` gets you another 30вҖ“40% plus tree-shakable code once you modularize.
- рҹҹ  **Firebase compat SDK.** See category 3. Large; switch to modular.
- рҹҹЎ **`querySelectorAll` in hot paths.** Many animation / powerup handlers re-query the same cells. At 4Г—4 this doesn't matter, but code elsewhere (sunk animation, laser highlight) queries across all boards on mouseenter. Cache.
- рҹ”ө Timer intervals (`startGameTimer` `:2963`, turn timer `:6377`) use `setInterval` with 1-second tick for a *display* вҖ” fine, but `requestAnimationFrame` + `Date.now()` delta is cleaner and handles tab-throttle correctly.

### 10. Responsive & Mobile

**Strengths:**

- `--cell-size: 42px` + CSS grid means boards scale as a block. Four media query breakpoints (1024 / 768 / 600 / 480 / landscape) cover common phones and tablets.
- Mobile web app meta tags (`apple-mobile-web-app-capable`, `mobile-web-app-capable`) set.

**Issues:**

- рҹ”ҙ Touchmove rotates on any swipe (Top-10 #6).
- рҹҹ  **Touch targets on the 4Г—4 grid.** At `--cell-size: 42px` on a 375px-wide phone, with 4 layers side-by-side or stacked, cells can become uncomfortably small. The 44px minimum target is visible in one place (`:659`). Verify every layer on iPhone SE and Pixel 5 real sizes; right now the landscape breakpoint (`:1348`) exists but I didn't test its output.
- рҹҹ  **No safe-area inset support.** Notched phones will clip the header / controls. Add `env(safe-area-inset-*)`.
- рҹҹЎ The ship counter + commentary + header controls + timer stack vertically on narrow screens; the combined height can push the actual boards below the fold, forcing scroll while playing. Measure on real devices.

### 11. Security

Covered in detail in category 3. Additional items:

- рҹҹ  **Secrets posture.** Firebase API key is a public identifier (not a secret by Google's model) вҖ” fine to commit. The *rules* are the gate. If a real key ever replaces the placeholder: (a) commit `database.rules.json`, (b) add a test harness with the Firebase emulator, (c) enable App Check to limit to your domain.
- рҹҹ  **Replicate token** is scoped correctly вҖ” lives in GitHub Action secret `REPLICATE_API_TOKEN`, never shipped to client. Good.
- рҹҹЎ **No CSP meta.** Add `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' https://www.gstatic.com; connect-src 'self' *.firebaseio.com wss://*.firebaseio.com; font-src 'self' https://fonts.gstatic.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com">` or similar. Blocks XSS escalation even if a peer message sneaks HTML through вҖ” currently `innerHTML` is used in 20+ places with interpolated values (e.g., commentary `<span class="notif-name">${powerup.name}</span>` вҖ” low-risk because `powerup.name` is a constant, but `data-mission-id="${mission.id}"` etc. all trust data).
- рҹҹЎ **Click-to-copy `document.execCommand`** fallback not needed (`navigator.clipboard.writeText` used) вҖ” flag is that some older mobile browsers won't have it. Fall back gracefully.

### 12. Testing & Reliability

**Issues:**

- рҹ”ҙ **Zero tests.** No test runner, no `package.json`. The `GameState` rules engine is the single highest ROI for tests in this codebase. Top-10 #10.
- рҹҹ  **No error tracking.** `console.error` in ~12 places (including inside `try/catch` that swallows), but no Sentry / Rollbar. Online desyncs or Firebase edge cases are invisible in production.
- рҹҹ  **No structured logging.** Mix of `console.log('Processing peer data:', data)` and nothing. Add a single `debug()` helper gated on `localStorage.getItem('debug')`.
- рҹҹЎ **Reproducibility:** see category 2 вҖ” without determinism, repro-from-log is impossible.

### 13. Product & Strategic Direction

**Observations on the twists:**

- **4 layers** вҖ” is this actually different? In practice it's 4 Г— 4Г—4 boards attacked independently. The coordination opportunity is thin because ships stay in their own layer (Spacecraft вҶ’ Space, Battleship вҶ’ Sea, etc.). The *rule* that enables coordination is the Krypton Laser cross-layer attack вҖ” the single twist that uses the layered structure meaningfully. Consider more cross-layer mechanics: spotter ship in one layer reveals adjacent layer cells, boarding action after sinking moves a ship down one layer, etc. Otherwise the layers are mostly a 4Г— multiplier on a 4Г—4 Battleship.
- **Power-ups** вҖ” BlackBox (adds a ship), KryptonLaser (cross-layer hit), CannonBall (2Г—2 area). These are fine differentiators but ALL gated behind a single random treasure chest per match. The chest is placed randomly on the 16-cell Sub board, so finding it is ~6% per Sub shot. That's a lot of variance for a match-defining advantage. Options: two chests, or a steady "treasure score" earned by hits.
- **Campaign modifiers** вҖ” these are the actual differentiator and they're undersold. A quick landing-page shot of the Kraken mission with shields + reduced fleet would do more than any UI polish.

**Half-built / abandoned:**

- `loadSavedGame` without a save.
- `Tutorial` class constructed but never started.
- `showSonarEffect` still exists from the SonarPulse вҶ’ CannonBall rename.
- Statistics class tracks `quickestWin`, `timePlayed`, `hitAccuracy` but the lifetime stats view is never shown.

**Missing (worth building, in order of impact):**

1. **Daily challenge mission** вҖ” one mission a day, same seed for everyone, leaderboard by accuracy. Leverages the campaign engine you already have; changes the posture from "I beat the campaign" to "I play this daily".
2. **AI-vs-AI spectator mode** вҖ” trivial given the existing AI, demos the game in background on landing page, also serves as a regression test for the rules engine.
3. **Fleet editor** вҖ” let players pick their 5 ships from a pool (adding Destroyer, Carrier, Minesweeper). Already half-present via `extraShips` in campaign.
4. **Replays / share-a-match** вҖ” once you have determinism + seeded RNG, a URL hash encoding the match lets players show off wins.

**AI integration opportunities (non-gratuitous):**

- The **portraits-via-Replicate** pattern is the good version: content you'd never hand-draw, gated behind a manual workflow, committed as static assets. Keep doing that.
- **Enemy briefing variations** вҖ” you already have prompt-written mission briefings. Generating a short pre-match trash-talk line per enemy (3вҖ“5 variations, committed) would add flavor.
- **Dynamic difficulty** вҖ” not an LLM use case; a boring linear regression on past-mission accuracy would work fine and not require an API call. Don't force AI where a number works.
